聚合搜索平台

介绍

这不是一个简单的搜索功能的那种项目，而是打算做成那种通用的，搜索平台的功能。 比如B站我们通过搜索框能够搜出来的是不限于某个板块的，而是比如说搜索Java，会检索出视频、用户、直播、文章等等这里面包含的所有和Java有关的内容。

并且这样的好处就在于不需要针对每个板块都做一个搜索功能。

但是我们还是会从单一搜索开始，最后在进行聚合。

初始化

初始化前后端就不做赘述，我们先去获取一些不同类型的数据资源。

这里获取三种不同的数据源：

  • 文章
  • 用户
  • 图片

针对这个页面，我们可能会出现以下几种业务场景：

  • 点击每个不同的tab的时候触发一次不同的查询，用户点那个tab就触发哪个
  • 针对内容聚合的页面，可以只发一个请求，后把数据放到每个tab里面。

这是这个项目主要要实现的，对于这个内容聚合的页面，可以这么理解：就是这个界面包含了很多的tab，并且这些tab的数据都是在一个页面直接展示出来了的，没有经过手电点击切换。

并且当tab数量变多了之后，第一种业务场景，前端需要写大量的请求，并且每个向后端请求的字段名是不一样的，可能会带来较大的沟通成本。

实现方式：

实现聚合主要就是解决：前端接口太多、前后端字段名不一致的问题。

所以自然而然的方法就是，前端统一一个接口，传递一个参数，到后端来完成不同tab的查找，最后将数据拼在一起返回给前端。

问题：那这样的聚合的搜索怎么完成分页？

  • 如果是所有tab放在同一个页面里的，其实这样可以不用去实现分页。当然也可以参照第二种的方式去实现

  • 如果是不同tab，根据点击来切换的，可以让前端每次传递的时候，会携带一个 type，让后端来辨别出同一个请求我要返回的数据类型是哪一个，然后调用对应的service，就可以完成对这一个type的分页。

    对于第一个场景，我们可以传一个 type的list，将list里面所有的内容都调用不同的service的时候分别完成分页。

实现聚合

其实上面说了那么多杂七杂八的，聚合实现起来非常的容易，无非是从多个接口变成一个接口，多个返回值变成一个返回值。

那不就是重新定义一个对象，里面接收不同接口的List 数据吗，然后在统一接口中，分别将数据查出来，然后设置到对象里面，返回给前端就行了。

经过测试发现 使用并发不一定更快了。

修改之前，我们之前也说到，我们可能会要在某个tab里面做分页，但是我在这个tab里面做分页之后又不想别的tab的页数发生改变。只是上面那样的接口的话，如果切换到一个新的页面，所有的搜索结果都是会发生改变到第新的页数的。

所以我们上面说到可以让前端多传一个 tpye来标识当前查看的是哪个页面，单独对页面就行分页。

上面这样添加一个tpye来确定具体要查询的对象，其实是用到了设计模式中的一种：门面模式。

这种设计模式就是类似于到银行办理业务，你不需要到每个窗口去看有没人，能不能办理这个业务，而是通过一个大堂经理他会告诉你去哪个窗口(假设我们这里的比喻不需要拿票)，你知道去哪个窗口之后就可以直接去。而我们这个系统就是前端我根本不关心后端怎么实现，我只要告诉你我要那个 type的数据，后端就会直接给我返回这个type的数据。

代码层面优化

但是看到这里是不是会有一个问题，如果我后面系统搜索的数据源扩展了，比如说我要加上能够搜索视频、网站等等。那我岂不是要写一大堆 service，调用不同的service明明干的都是差不多事情。

所以我们需要使用设计模式来及解决这样一大堆的重复的代码。

​ 首先我们先思考以下什么样的数据能够接入我们的系统，我们是不是应该定义一套规范。我们总不能让用户随便接入吧，然后我们再根据他们接入的代码来改我们自己的代码，这显然很不合理。

​ 所以我们需要定义一套规范，让所有接入数据的都实现这套规范，而定义规范最好用的不就是接口了嘛？

​ 我们指定要接入我们系统的数据都必须要实现我们指定的接口，并且重写里面的方法，我们这个项目比较简单，我们就只要规定每个接入的数据源都要提供一个支持分页搜索的方法供我们的系统调用。并且我们将我们上面已经实现的自己的数据源也按照这样的规则进行修改。

数据同步

和Redis一样，使用Es同样会出现数据不一致的情况，所以需要实现数据同步：

实现数据同步主要有四种方式：

  • 全量同步(一般是项目启动的时候)，将mysql中的数据全部同步到Es中，但是项目中也是可以使用全量同步的，只不过可能会开销较大，每次同步所有数据，用定时任务的方式。

  • 增量同步，指的就是启动之后有新增数据之后要进行数据的同步，全量和增量一般都是通过定时任务的方式来实现的，但是为了防止同步失败，并不会之在mysql中查找一个定时周期内的数据，而是会同步定时周期的2-3倍。比如说 一分钟同步一次，前一次同步失败了，如果还是同步前一分钟的，就可能会导致丢失前一分钟的数据。所以一般会查找2-3分钟前的数据进行同步

上面两种方式的使用场景：

对于那些实时性要求不强的数据可以使用，数据量不大的系统也可以使用。

优点就是简单、不需要第三方工具

缺点就是实时性较差

  • 双写：写数据库的时候也写如ES，但是这样的问题就是可能会出现事务问题，可能es跟新失败，所以需要配合日志、告警等方式来实现数据补偿
  • 第三方工具：logstash(定时同步，只是把我们的编码内容配置化)、canal(实时同步)
