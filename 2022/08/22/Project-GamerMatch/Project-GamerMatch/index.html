2、主动搜索：根据标签搜索其他的用户

3、组队（创建房间/队伍）：

4、推荐系统

现在有两种做法让用户表和标签表扯上关系：

1、User 表新增字段

2、新建关联表，做userId到tagId的映射

如何选择？

第一种方式的优缺点：

  • 优点
    • 不需要新建一个表
    • 查询比较方便，查询用户的时候就已经把标签带出来了
    • 并且如果我们以 Json字符串的形式同样能够很好的满足1对多的场景，同时根据标签名查找的时候可以直接用 模糊匹配的方式 匹配字符串来实现‘
    • 能够节省开发成本
  • 缺点
    • 在大数据量下可能 like 匹配速度较慢，但是我们可以通过缓存来优化，因为标签也是属于不经常变更的属性，就算变更造成不一致对结果影响不算特别大

第二种方式的优缺点：

  • 优点：
    • 大数据量下的查询效率也能得到一定保证，配合缓存效果更佳
    • 关联表能够很容易的实现反向查找，比如说获得标签名下的所有用户，通过连表就能够解决
  • 缺点
    • 实现起来复杂，并且数据量一般不会很大没必要搞得这么复杂
    • 并且关联查询需要避免大量使用，在阿里巴巴规范中不建议超过三个表以上的关联，影响拓展性

综合考虑在当前项目中使用第一种做法

大批量插入数据

这也可以当作是一个面试题，就是如果要你导入1000w条数据你会怎么操作？

1、暴力for循环，for(0-1000w)

自己的项目可以这么干，但是实际中肯定不行。因为写数据库比较耗时，况且还是大批量的写入，可能要花费几十分钟甚至一个小时以上，那么这时候数据库压力就会非常大，可能导致数据库无法查询，从而导致用户无法使用接口。

最重要的是这种插入会导致系统变得不可控，如果运维要你停止插入，你要到机器中终止这个进程，终止这个进程可能还会导致原本项目部分接口不能使用，所以非常危险直接暴力插入。

  • 做十条数据的测试

10条数据花费了273ms，如果1000条就要30s，1000W就是30Ws，5000分钟？？

其实我们可以看到每次插入数据都要建立连接和关闭连接，这一块的开销是非常大的。

2、分批插入

分批插入数据，甚至可以用上异步任务或者多线程处理

使用缓存，但是第一个用户还是访问比较慢，如何解决？

使用缓存预热，将部分数据先加载到Redis中。缓存预热是个不错的做法但是有什么缺点？

1、增加开发成本，需要确定好预热的数据和时机

2、占用Redis空间，如果正常运行的时候Redis已经快满了，还要预热数据就很容易造成Redis OOM。

缓存预热

怎么实现缓存预热，有哪些注意点。

  • 只加载一次不需要跟新的：项目启动后自动加载，实现CommandLineRunner接口，实现run方法。
  • 可能会需要跟新的：定时任务，

缓存预热需要注意哪些：

  • 预热的时机和数据，不能太多不要占过多的空间，一般把热点数据加进去就行了

  • 数据是否需要跟新，防止预热的数据太老，跟新的频率和时机，不能在用户活跃的情况下进行跟新。

  • 对于分布式项目，整个项目中同一个定时任务，只有一个服务器执行真正的逻辑，因为如果一个项目部署到了多台服务器，每个服务器都到期会执行一次定时任务，如果这个定时任务需要大规模的读写数据库，那岂不是可能会操作数据库被我们自己搞死了。并且对于写数据库，还可能会出现写入重复的数据。怎么实现？

    • 分离定时任务，把定时任务单独部署。太麻烦，成本太高
    • 加上配置判断，比如通过一个配置，只允许一个IP执行真实逻辑，其他的都直接返回。
      • Nacos
      • Redis
      • Mysql
    • 分布式锁，setnx简单实现、hash结构增加可重入性、rediss直接实现。 使用分布式锁的时候可能还会出现一种情况，就是如果Redis也是集群配置的时候，会不会出现加锁出现数据不一致的情况？

    答案显然是会的，因为Redis只要主节点完成写入就会向用户返回写入成功，然后重启线程开始主从复制，所以可能在复制过程中出现异常，比如主节点宕机，然后从节点上位，造成数据不一致。

    为了解决这样的问题，Redis提出了一种红锁算法，就是部署的都是Redis主节点，并且写锁的时候只有大于1/2的节点写入成功才算获得锁，从而尽量避免了主节点宕机出现的各种问题。

实现简单的curd这里就不列出来了。但是有一个问题，就是为什么有些时候我们要封装好 Dto、Vo？

Dto：

一般是对前端请求后端的参数的封装，

  • 请求的字段的名/类型和实体类可能不一样
  • 有些参数用不到，降低理解成本

Vo：

一般是后端返回给前端的数据的封装，

  • 除了上面的两个类似的问题以外
  • 还有一个更重要的就是在于有些数据对于前端是需要脱敏的，不能够返回给前端让前端或者用户看到

那有了封装的对象，我们难免会经常要进行Dto–>实体类，实体类–>Vo的映射，总不可能我们重复写set、get？

我们可以使用Apache的一个BeanUtils工具类帮我们完成。

怎么匹配？

取出所有用户依次匹配tags，太暴力性能差。

能做哪些优化？

  • 对于某些数据结构，可能会存储很多没用的内容，比如 map在这里我们用作存放每个用户和当前用户匹配计算的距离，但是实际上我们可以只需要存放前五或者前几的用户，减少比较大的内存开销。
  • 查询全表的时候，我们尽量只查找我们需要的几个字段，而不是一股脑的select *
  • 先剔除不可能满足条件的用户
  • 提前先查找好，然后放缓存，用户需要的时候直接从缓存获取，提升用户体验
  • 还有一个就是，在大量的SQL查询的时候一定要关闭日志输出

下面几个方法的注解就是用于触发特定的事件的。

@ServerEndpoint，有一个比较坑的地方，，这个类被视为一个 WebSocket 端点（Endpoint），会让自动注入失效，因为该注解是 Java API for WebSocket 提供的，用于处理 WebSocket 连接的创建和管理。WebSocket 的生命周期由 Java API for WebSocket 管理，而不是由 Spring 管理。虽然同时加了类似于@Component的注解，但是@ServerEndpoint优先级更高，因此，Spring 的依赖注入机制不会生效。

解决办法：

其实这种做法是一种通用的解法，通过配置我们自己的SpringContext，在自动注入失效的类上通过手动注入。

  • SpringContextUtils，用于手动获取bean，实现ApplicationContextAware接口，接口提供了一个方法 setApplicationContext(ApplicationContext applicationContext)，**当一个 bean 实现了该接口并被 Spring 管理时(也就是说我们在使用的时候，这并不是一个普通的utils对象，而是同样需要被Spring管理的对象)**，Spring 会在 bean 初始化过程中自动调用该方法，并将 ApplicationContext 的引用作为参数传递给该方法。

有了这个bean，在无法完成自动注入的类中，可以直接通过这个类所提供的方法获取到spring管理的bean。
