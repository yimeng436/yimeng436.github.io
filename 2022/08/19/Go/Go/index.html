扩容

slice扩容机制在1.17之前版本和1.18之后发生了一点变化：

  • 1.17之前：

会先对新的slice对象的容量是否大于原来的两倍，如果大于就直接设置新的容量，否则会判断旧的容量是否大于1024，如果不是就直接设置为原来容量的两倍，如果大于1024则会反复增加原容量的1/4，直到满足新容量。

  • 1.18之后：

直接判断新的容量是否大于旧容量的两倍，如果大于直接等于新的容量。否则不直接和1024做对比，而是和256在做对比，当新的容量小于旧容量的两倍时候。会判断旧容量是否大于256，如果小于256就直接将新的容量设置为旧容量的两倍，否则会按照一个特定的扩容公式进行扩容。

1.17之前的扩容机制在容量大小小于1024的时候会根据2倍进行扩容，当大于1024时会根据1.25倍进行扩容。这样做的好处是，当容量大的时候不至于扩容太过分，导致很多空间无法被使用，以1.25进行扩容较为平缓，不会一次性增加过多的新空间。

但是1.17在1024之前的扩容速率是2，这样的扩容速率不够平缓，在低容量的时候同样可能会出现一次性增加很多空闲空间。所以在1.18引入新的扩容公式主要就是为了解决小容量扩容速率太大的问题。让容量 在256-1024的切片扩容速率都是小于2的，更容易被利用。

除此之外，还要注意的是，切片发生扩容时候，由于底层还是数组，长度还是无法改变的，所以发生扩容时，会将原数据拷贝到一个新的数组里面，再追加元素，也就是说，发生扩容时，扩容前后的切片对象还是同一个地址相同，但是他们底层的数组不是同一个，可以通过打印第一个元素的地址查看。

这一点很重要，如果是函数形参发生扩容，那扩容后再对切片的元素进行操作将无法影响到实参。

逃逸分析

在Java中，没有指针的概念，数组作为参数时候实际上传递的是引用。

但是在go中，数组是一个值变量，每次作为参数传递的时候，会对原数组进行拷贝，所以我们可能会想，如果每次都拷贝一个数组这样的系统开销是不是很大，并且go中是有指针的，使用指针传递数组是不是可以避免每次进行拷贝数组，从而减少系统开销呢。

在数组数据量非常大的时候，这么做是一个很不错的选择，但是一般情况下由于go中有灵活的slice，所以数组使用频率不会特别高，更别说大数组了。那也就是说通常情况是一个小数组，如果使用指针来传递的话，可能会出现效率反而下降的情况。

我们都知道，正常函数都是运行在栈中的，函数结束，栈自然释放，不需要涉及到 gc。

相反堆中的数据是不同于栈中的，会一直存在，直到被gc回收。而启动gc是一个非常消耗系统资源的一个动作。

通常我们定义的变量不是在栈里，就是在堆里，因此我们要避免将数据定义到堆中。

go的编译器会自动帮我们分析我们定义的变量需要被存放在什么地方，一般满足以下两个规则：

  • 函数外部没有引用就优先放到栈中
  • 函数外部有引用就一定会放到堆中

因此回到上面一开始的问题，如果是指针传递，外部的数组一定是存在的，所以指针会被创建到堆中，函数结束之后不会被释放，要等待gc 所以可能会导致系统开销更大。

哪些情况会导致逃逸？

通常来说有以下几种情况：

  • 函数返回的是对象的引用，那么就会发生逃逸

可以通过以下指令来分析，变量是否逃逸

  • 数组、切片长度分配过大也会导致逃逸
  • 创建切片时候使用变量，编译器无法知道具体的长度，也会在堆中创建，也可能会出现逃逸。

map

map 底层由两个结构体构成，一个是 hmap，一个是 bmap。

hmap负责存放一些基本信息，如k-v对的个数，bmap的个数，指向bmap的指针等

bmap俗称为是桶，主要用于存放 k-v的结构体。

整体的结构如下：

初始化

会发生什么？

1、首先会创建一个hmap对象，初始化计数器为0

2、根据期长度，计算B的值，会根据一定的计算方法得到B，默认0

3、利用计算得到的B，去创建2 ^ B的桶

4、生成一个哈希因子，用于计算key的hash值

这里在创建桶个数的时候其实还会有一个对B的限制：

  • B<4 直接创建 2B 个桶，这个桶被称为标准桶
  • B>=4 会创建 2B + 2B-4 这个时候桶就是标准桶+溢出桶，这个溢出桶其实有点像操作系统的二级分页一样，一个bmap结构体会有一个指向下一个bmap的指针，此时后面的就是溢出桶。

写数据

1、首先会根据key和哈希因子计算得到一个hash值。

2、根据这个哈希值的 最后 B 位，确定要存放到哪个bmap里面，也就是确定放到哪个桶里面

3、确定好了放哪个bmap之后，我们看到bmap的主要结构，有四个属性

  • topbits 存放的是key的hash值的高八位，用于快速对比，如果高八位都不相同那么这个key就更不可能相同了，和 Java 先比较hash值，再比较key一个道理
  • keys 存放key
  • values 存放value
  • overflow 指向下一个bmap

从这个定义我们也可以看到一个bmap只能存 8个 k-v，并且go不是直接以 k-v对存储的，而是用数组， 三个数组的下标0-7一一对应一个k-v对

读数据

和写数据差不多是一个相反的过程。

根据key和哈希因子，计算得到一个哈希值，取后B位得到这个key在哪个桶。

然后取key哈希值的前8位和这个桶的topbits数组一一比较，如果对比成功就去key的下标里面对比key是否相同，如果相同取出下标对应的value。

如果当前桶没有匹配的key，会向下查找溢出桶。

  • hmap的count（k-v的总个数）/桶个数（2B  ）> 6.5时候发生扩容，翻倍扩容
  • 溢出桶太多时会发生扩容，具体数据如下：
    • B<=15，一使用的溢出桶个数>=2B ,引发等量扩容，这个等量扩容就是，扩容前后桶个数不变，只是数据会发生迁移，具体的迁移方式，在后面讲
    • B>15，已使用的溢出桶的个数>=215  ,引发等量扩容。

这里需要看一下详细的 hmap的结构：

有几个字段和扩容有关系：

flags：判断是否正在写入，并发时候使用

oldbuckets：扩容之后指向原来的桶的指针，这里只发生了扩容，新桶是没有数据的，还要等数据从旧桶中迁移到新桶中，所以需要保留旧桶的指针

nevacuate：下一个要发生迁移的桶

extra：是另一个结构体的指针

这是管理溢出桶的一个结构体，overflow指向的是溢出桶数组的指针，oldoverflow是扩容时，指向旧的溢出桶s数组的指针。nextOverflpw是指向下一个空闲溢出桶的指针。

由于扩容分为等量扩容倍增扩容，所以在数据迁移的时候也会有些不同。

  • 倍增扩容

假设原来是4个桶， 那扩容后应该变为8个，所以对应的B值就应该加1，在迁移时，会在旧桶中遍历所有的key包括溢出桶的key，然后重新计算他们的哈希值，然后取 后B+1位，就是这个key的新的存放桶。

  • 等量扩容

等量扩容就只是，讲原来的桶存到新的桶的相同的位置，这样做是为了让那些旧桶中的元素更为紧凑，因为可能旧桶中存在很多的删除后的k-v。

协程(goroutine)、管道(chanel)

在go中，一个线程可以启动多个协程，协程就是一个轻量级的线程。 协程必须在主线程执行结束之前结束，否则主线程结束会让协程也一起结束。

协程比线程更低一级的运行方式，它不属于操作系统功能，不由操作系统分配cpu、内存，而是由golang自己实现运行资源的调度使用。也正是因此执行效率要高于线程。

相比于Java这样的多线程并发编程语言，go采用的协程相比于线程占有的资源较少，上下文切换花费时间较少，速度更快。

goland 中用 go关键字去开启一个go协程。

执行上面的程序会发现，开启协程的那个方法根本没有执行。为什么？

这个需要结合go语言并发的模型，MPG模型

  • M（Machine）：操作系统的主线程

  • P（Processor）：协程执行需要的资源（上下文context），可以看作一个局部的调度器，使go代码在一个线程上跑，他是实现从N：1到N：M映射的关键

  • G（Gorountine）：协程，有自己的栈。包含指令指针（instruction pointer）和其它信息（正在等待的channel等等），用于调度。一个P下面可以有多个G

一开始运行go就会创建一个主协程(线程)，执行到go语句时候，就会创建一个调度点，这个协程就会挂在P的队列下面。当主协程被阻塞或者等待时，就会在P的队列下面取出一个协程进行执行。

这样也就很好解释为什么上面的代码 sayhello方法为什么没有被执行了，因为主协程是一直执行的，没有被阻塞，所以没有调度go 开启的协程，又因为协程必须在主协程执行结束前调度，所以 sayhello不会被调度。

如果在主协程上加上一个 sleep就能够调度到 sayhello函数。

和Java 一样，这样写死等待时间的方式是不可取的，类比Java 当时解决这个问题采用的是countDownLanch，go种也有类似的实现， go中使用的是sync包下的一个结构体WaitGroup，开启协程的时候计数器加一，协程任务结束的时候调用Done函数。
