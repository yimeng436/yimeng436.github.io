扩容

既然是动态数组，可定会涉及到扩容的问题。扩容只会发生在append时候。

根据以下规则进行扩容：

  • 如果新的切片长度大于旧切片容量的两倍，直接将新切片容量就等于新切片的长度

  • 否则，会进行以下判断：

    • 旧切片容量小于256，新切片容量直接变为旧切片容量的两倍
    • 反之则每次以旧切片容量的1.25倍进行扩容，直到大于新切片长度

  • 最后会按照类型占用的空间，以及元素的长度，去计算扩容到新容量后所占用的内存，并且会进行内存对齐，go是按照块进行内存分配的，所以

作为函数参数

在go语言中，数组不是一个引用数据类型，作为函数进行传递的时候，形参实际上是对数组的一个拷贝。函数中的操作不会影响原数组的内容。

但是go语言在进行参数传递的时候只有值传递，也就是说，形参都是对于实参的一个拷贝，那Slice作为引用数据类型，作为函数参数会怎么样？

首先，函数接收的是slice的一个拷贝，但由于Slice底层有一个指向数组的指针，这个指针也会被拷贝到形参中，因此，函数中的修改操作可能会影响到原Slice

Map

首先我们先要明确以下，go中的map是通过拉链法来解决hash冲突的。

拉链法就是利用数组和链表，key冲突了，把相同hash值的key挂在数组下标的链表下。

还有比较常见的解决hash冲突的方法有开放地址法，就是当发生冲突会根据某些策略继续往后面进行探测，知道没有发生hash冲突。

基本结构

创建一个map，底层的数据结构实际上是 hmap。

主要存数据的就是 buckets，这是一个指向bmap数组的指针。在图中也可以看出来，除了buckets以外，oldbuckets也是指向bmap数组的指针。

这个bmap就是实际存储数据的结构，

overflow 指向的就是溢出桶，也就是出现hash冲突存放的bmap的地方。

比如说：

图上那的绿色的bmap的结构，假设他是buckets[0] 对应的bmap。当绿色这个bmap的8个key，value都存满了的话，此时又到了一个key-value，也是想存在当前这个buckets[0]的bmap下，此时就会把这个k-v存到overflow指向的bmap里面。

初始化

会发生什么？

1、首先会创建一个hmap对象，初始化计数器为0

2、根据期长度，计算B的值，会根据一定的计算方法得到B，默认0

3、利用计算得到的B，去创建2 ^ B的桶

4、生成一个哈希因子，用于计算key的hash值

这里在创建桶个数的时候其实还会有一个对B的限制：

  • B<4 直接创建 2B 个桶，这个桶被称为标准桶

  • B>=4 会创建 2B + 2B-4 这个时候桶就是标准桶+溢出桶，这个溢出桶其实有点像操作系统的二级分页一样，一个bmap结构体会有一个指向下一个bmap的指针，此时后面的就是溢出桶。

写数据

1、首先会根据key和哈希因子计算得到一个hash值。

2、根据这个哈希值的 最后 B 位，确定要存放到哪个bmap里面，也就是确定放到哪个桶里面

3、确定好了放哪个bmap之后，我们看到bmap的主要结构，有四个属性

  • topbits 存放的是key的hash值的高八位，用于快速对比，如果高八位都不相同那么这个key就更不可能相同了，和 Java 先比较hash值，再比较key一个道理
  • keys 存放key
  • values 存放value
  • overflow 指向下一个bmap

从这个定义我们也可以看到一个bmap只能存 8个 k-v，并且go不是直接以 k-v对存储的，而是用数组， 三个数组的下标0-7一一对应一个k-v对

读数据

和写数据差不多是一个相反的过程。

根据key和哈希因子，计算得到一个哈希值，取后B位得到这个key在哪个桶。

然后取key哈希值的前8位和这个桶的topbits数组一一比较，如果对比成功就去key的下标里面对比key是否相同，如果相同取出下标对应的value。

如果当前桶没有匹配的key，会向下查找溢出桶。

扩容

和Java的 HashMap 一样，在一定条件下map也会发生扩容：

  • hmap的count（k-v的总个数）/桶个数（2B  ）> 6.5时候发生扩容，翻倍扩容
  • 溢出桶太多时会发生扩容，具体数据如下：
    • B<=15，已使用的溢出桶个数>=2B ,引发等量扩容，这个等量扩容就是，扩容前后桶个数不变，只是数据会发生迁移，具体的迁移方式，在后面讲
    • B>15，已使用的溢出桶的个数>=215  ,引发等量扩容。

这里需要看一下详细的 hmap的结构：

有几个字段和扩容有关系：

flags：判断是否正在写入，并发时候使用

oldbuckets：扩容之后指向原来的桶的指针，这里只发生了扩容，新桶是没有数据的，还要等数据从旧桶中迁移到新桶中，所以需要保留旧桶的指针

nevacuate：下一个要发生迁移的桶

extra：是另一个结构体的指针

这是管理溢出桶的一个结构体，overflow指向的是溢出桶数组的指针，oldoverflow是扩容时，指向旧的溢出桶s数组的指针。nextOverflpw是指向下一个空闲溢出桶的指针。

迁移

由于扩容分为等量扩容倍增扩容，所以在数据迁移的时候也会有些不同。

  • 倍增扩容

假设原来是4个桶， 那扩容后应该变为8个，所以对应的B值就应该加1，在迁移时，会在旧桶中遍历所有的key包括溢出桶的key，然后重新计算他们的哈希值，然后取 后B+1位，就是这个key的新的存放桶。

  • 等量扩容

等量扩容就只是，讲原来的桶存到新的桶的相同的位置，这样做是为了让那些旧桶中的元素更为紧凑，因为可能旧桶中存在很多的删除后的k-v。

需要注意的是，go的map采用的是写时复制的方式，所以一开始的新的buckets 不存在数据，并且如果后面map一直被读，就不会发生数据写入新的buckets。

删除

在删除map的key的时候，除了删除掉key和value 以外，还会将tophash对应的下标改为emptyOne，如果后面的槽位都不存在元素就会改成emptyRest，往前判断，如果前面一个也是emptyOne，把前面的也设为emptyRest。

问题

和 Java一样，go的 map也是并发不安全的，都是除了读读可以并发以外，写写、读写都会直接爆出panic，并发修改异常。

所以在读数据，写数据的时候一开始都会判断是否正在写入，只有不是在写入的状态才能够读、写。

map每次遍历的结果都是不一样的，这是因为map每次 遍历的时候都会随机从一个bucket开始，进行遍历。并且在bucket中的槽位也是随机开始的。

这样设计的原因有以下两点：

1、map 会发生扩容，扩容不是原子操作。如果遍历的时候发生的扩容，这两次遍历的结果本来就可能会出现不同。

2、同一个元素，先删除再插入，前后的位置可能都是不一样的。

所以理论上，map的遍历结果本身就应该是不同的，所以为了防止用户依赖遍历的顺序，就设计了这样的方式。

写数据的时候还有一个要注意的就是，判断完是否处于写状态之后，会将状态改为写状态。

在最后会有一个写数据 收尾的操作。

他会再判断一次是否处于写状态，如果是就抛异常，很典型的乐观锁的实现方式。再清楚写状态。

如何实现有序的map遍历？

把map的key拿出来，保存到一个slice中，对slice进行排序，每次遍历的时候，再根据slice中key的顺序进行获取value。

原生map是否是并发安全的？

不是。

在golang map的源码中，hmap结构有一个flag字段，用于表示map的状态。在执行写操作的时候，会修改这个字段，但是并没有同步措施保护这个字段，所以在多线程操作的时候，就可能会出现数据的不一致性，导致数据错乱，甚至程序崩溃。

为什么要这样设计？

这样设计的目的有以下几个原因：

性能考量：map要设计成并发安全的话，需要经常加锁解锁，会带来较大的性能开销。并且go设计的初衷就是简单高效的语言特性，它更倾向于，让程序员根据情况决定是否加锁，加什么锁，比如sync.Mutex、sync.RWMutex，或者用channel来实现同步访问或者修改map。这一点也非常体现了，go语言的高灵活性的设计目的。

另外，大部分情况都是不需要考虑到并发安全的，如果统一对map实现并发安全，会对不需要并发控制的程序带来不必要的开销。

sync.Map

由于 原生 map 不支持并发炒作，所以go提供了一个支持并发的map，核心是利用空间换时间的方式来实现的。

它包括两个map，一个是read map 另一个是 dirty map。

举一个不恰当的例子，这个read map 和 dirty map 非常像我们的redis 和 mysql的关系。

我们除了写以外的操作都是尽量在redis(read map ) 上进行，然后如果对于修改删除，最后会同步到**mysql(dirty map)，而mysql(dirty map)就相当于是redis(read map)的一个兜底，所有操作都可以在这里完成，因此mysql(dirty map)**拥有全量的数据。

同样redis我们会尽最大限度访问，read map同样可以尽最大限度访问。

而mysql我们不希望被大量的访问，因此dirty map也有最大限流操作，也就是加锁操作，同一时间只有一个 操作 能够在dirty map上执行。

有一点不同的就是 dirty map 和 read map 有一个数据共享的机制，相同的key指向的都是同一个value的内存空间。

基本API操作

结构

dirty map 有互斥锁来保护，read map没有用这么强的锁进行保护，但是他也并不是并发不安全的，而是通过CAS操作来实现。

双向数据流转机制

既然有两个存储数据的容器，那么肯定需要有某种数据流转的方式，才能让两个map 都发挥作用，不至于所有数都找dirty map，或者都找 read map。

read map ==>dirty map 全量数据的**copy**操作，时间复杂度是 O(n)，需要遍历read map 里面的所有key。

dirty map ==> read map 将全量数据**直接**赋值给read map，O(1)，赋值语句。

数据流转出发时机

当大量操作都是在 dirty map上完成的，也就是 miss 一直在增加，在达到一定阈值的时候，就会触发dirty map ==> read map

上面也说到了，这种情况是，dirty map直接将数据赋值给read map，其实也不算是赋值，赋值完之后，dirty map的所有值都会清空，直接置为nil。

那既然dirty map 都为nil了，此时就无法完成兜底的工作，所以第二个数据流转触发的时机，就是写入一个数据，read map无法完成时，就要将read map中的数据拷贝一份给 dirty map，并且在dirty map上再完成写入操作。并且这里需要注意的是，这里只会把read map中逻辑存在的那些key拷贝给dirty map，逻辑存在就是指能够读取到value的那些key。

看到这可能会出现的疑问就是，这种流转方式是不是多此一举，为什么dirty map要直接设置为nil，不能够赋值给read map，这样就避免了，read map的拷贝不是吗？

实际上，设置为nil 是有必要的，因为map是引用数据类型，如果只是简单的赋值，会导致read map 和 dirty map 不是相互隔离的，相当于他们共享了同一个map，这样就起不到隔离的效果。

一定不能够直接将dirty map的数据copy 一份给 read map。

如果是这样做的，我们上面对于read map说过，对于read map的操作是不加锁，通过CAS来完成的。但是如果是需要dirty map 向read map写入数据，那就必须要对read map 加锁，来保证并发安全。但是如果这个操作要让read map 加锁，其他的操作要再操作read map的话不久也得获取这把锁，才能够操作read map。所以如果是将dirty map 拷贝一份给read map的话，就会使得在写入read map的期间，整个sync.Map都是无法使用的

entry

从结构中可以看出来，sync.Map中，每个key对应的value 都是一个entry。

从结构体可以看出，entry就只是将指针封装了一下。

这个p指针，是一个可以指向任意类型的指针，就是对 unsafe.Pointer封装了一下而已。

这个p除了指向正常的一个数据以外，还有其他的状态。

  • nil： 软删除，也就是两个map的key对应的数据逻辑上不存在了，但是物理地址还是存在的，只是不能通过这个指针找到。换句话说，就是处于软删除状态的entry，他们的key还存在，并且是在两个map中都存在，用这个key去访问时，会得到一个nil值。
  • expunged： 这是一个特殊指针，硬删除，这个表示它的key只在read map上存在，在dirty map 上不存在。
  • 正常数据

为什么要这样设计，不直接让entry指向，key对应的value。

nil状态设计的目的

我们知道对dirty map操作是需要加锁的，所以希望尽可能的减少在dirty map上的操作。

这样设计最主要就是为了优化删除操作，如果不是这样设计，而是直接对一个key进行 delete操作的话，因为没有这样设计，没有p指针状态的改变，所以删除操作肯定是要上锁的，这就导致了，read map没有帮dirty map消耗删除的流量。

而采用了p状态改变这种设计方式，我们就可以让read map 通过CAS操作修改p的指针状态，从而完成无锁删除。

除此之外，对于删除后的key-value，它的key还是存在的，这就使得，下一次写入这个key，相当于是一个跟新操作，又可以采用CAS的方式修改p对应的指针。这相当于优化了同一个key先删后写的场景。

expunged状态设计的目的

在介绍，双向数据流转的时候read map数据copy到dirty map 的时候，只会拷贝逻辑上存在的key，也就是p指向的不为nil的数据，但是如果后面这些逻辑上不存在的key，被写入数据了，没有这个expunged的状态就会导致，dirty map没有所有数据。

所以在read map ==> dirty map的时候，会把那些逻辑上不存在的key指向的value的p指针设置为 expunged状态，在下次写入数据的时候 ，如果这个key对应的p指针状态时expunged的话，就会直接将这个key和entry复制给dirty map，并且在dirty map上完成写入操作。

在遍历的时候会用tryExpungeLocked函数来判断value的状态，也就是判断entry 指针p的状态。

channel

channel被设计用于完成 goroutine 之间的通信的。

go 语言实现并发的思想遵从的是：以通信的方式来实现共享内存，而不是通过共享内存的方式来实现通信。

channel和map、slice 一样，都是通过make来分配内存的，但是和slice他们不一样的是，slice、map他们通过make分配得到的内存可能是在堆也可能是在栈上的，因为有些可能是全局的，就是在堆里面；有些可能是在函数内定义的，就是在栈里面，会随着代码段的结束而释放的。

channel通过make创建出来的一定是在堆上的，也就是channel的生命周期和作用域不会被限制在一个函数中。

select

一般是用于配合channel来使用的，他是一个多路监听器，可以监听多个channel的操作。

select一次可以监听多个channel，只要有一个channel的操作成功就会结束。因此select看也可以分为阻塞形和非阻塞形的。

如果没有default操作，所有case都没有执行成功，就会阻塞挂起，这就是阻塞形的。

有default操作，再判断完所有的channel操作之后，就会执行default，退出。

这个每次case的判断都是随机的。这里再讲一下阻塞形的，如果所有case都没有完成，那么就会将当前goroutine加入到所有 case所包含的channel中的等待队列中。

对于唤醒，就只要select里面有一个case操作被channel操作唤醒之后，就需要把当前goroutine从其他channel的等待队列中剔除。

5、channel实现限流

思路：创建一个buf为根据要求计算出要处理请求的频率记L，创建一个长度为L的channel，每次处理请求之前都要从channel中读出一个元素，才能往下执行，否则一直阻塞。

思路：和上面的差不多，创建一个buf为1的channel，每次要进行操作的时候，要能够从channel里面读取到数据才能操作，以此来表示枪锁，否则一直阻塞。每次执行完一个操作后向channel写入一个数据，表示释放锁。

defer

defer 是用于让某些函数延迟执行的，主要是让这些函数在return 和 panic之后执行。因此defer只能用于修饰函数。

defer可以多次调用，每次调用都会生成一个_defer的结构体

每个defer都会被放到 _defer链表中，是按照头插法的方式进行插入，取的时候又是从头顺序取，所以是满足LIFO的规则。

编译器遇到defer的时候，会插入两种函数

  • 1、defer内存分配函数：deferproc（堆分配）或者deferproStack（栈分配）

这里可以看出，会优先是由开放编码的方式，如果不满足，在没有内存逃逸的情况下会使用栈分配，否则使用堆分配。这样可以更好的提升性能。

并不是一开始就有这么多种分配方式的，在go1.13之前，只有堆分配。堆分配，通过内存复用来实现，采用了内存池的思想。会先在本地和全局的defer缓存池里找一个可用的defer结构，否则在堆上创建。

go1.13之后为了优化defer的性能，会根据defer语句包含的参数是否包含堆上分配的对象的引用来判断是创建在堆上还是在栈上。

go1.14之后，会根据defer函数个数是否大于8个或者defer函数在循环体内，如果满足这些要求就会分配在堆上。所以要尽量避免for循环内使用defer。

  • 2、执行函数：deferreturn

这个函数其实就是在return 关键字执行之后，会触发call调用这个函数，这个函数的逻辑就是，遍历goroutine的 defer链表，从头开始取，依次执行。

总结

  • 遇到defer会插入两个函数，内存分配函数和执行函数
  • defer满足LIFO规则，通过头插法链表实现
  • defer内存分配有三种方式：堆、栈、开放编码
