UserCenter

这是一个简单的项目，就只是实现用户的登录、注册、CURD。

知识点1，企业中对于用户的密码必须要加密处理

这里采用简单的MD5加密算法，这是一种单向的加密算法，只能够从明文==>密文

可以使用apache提供的工具类来实现

加密为了保证更难被破解可以通过 “加盐” 的方式，实际上就是加上一个字符串常量。

知识点2：实际写接口的时候一定要考虑权限问题，防止接口被恶意调用

简单的系统只需要通过 用户的角色字段来判断就行，登录的时候会将登陆的用户存到浏览器的session中，所以可以直接通过 HttpServletRequest来取登录用户的 role。

知识点3：数据脱敏，后端返回数据时候需要对敏感数据进行过滤，且要保证过滤逻辑一致

要保证一致所以可以通过定义一个 工具类 ，或者通过创建 Vo对象来实现脱敏。

所有查询都需要实现脱敏，所以需要对每个查询结果都经过我们的工具类或者转为Vo对象。

  • 工具类脱敏

知识点4：自定义异常以及全局异常处理

别的项目也用过，只是在这里再提一嘴。

自定义异常目的是能够提供更多的异常信息，包括状态码、增加字段等。

做法就是继承RuntimeException，然后增加自己想要的功能或者字段。

全局异常处理：目的为了捕获后端运行时出现的异常，以及避免在前端获取到异常信息，因为有些时候异常信息会包括我们写的代码、类、报名等内容，提供系统的安全性。

知识点5：多环境

多环境是指在企业开发中，肯定不可能我们增加、或者修改完成某个功能就直接将项目上线，而是要经过开发–测试–预上线–生产这么一个或部分的一个过程。

开发(dev)：一般就是程序员在自己的本机用本地的数据库、redis、MQ等服务完成某些功能，依赖的都是本地的数据，并且真实数据肯定也不可能在开发的时候被访问。

测试(test)：同理测试肯定就有对应用于测试的一组数据库，让测试人员先测测系统的BUG

预上线(pre)：直到预上线才可能是连接真实数据库测试，看看是否有测试遗漏的问题。这个阶段所用的数据库和正式上线的其实是一样的，只是访问的一些域名可能不同。有点类似于体验服，让部分的用户进来测试

生产(prod)：就是用于实际使用的，并且有专门的人员来维护和分配资源。

多环境的优点：

1、区分了不同的阶段使用不同的数据，

2、能够很好的保证真实数据不会被意外修改，

3、并且又能够很好的调试，在开发阶段我们可以加上各种各样的日志，但是实际上线了这些日志肯定是不能给前端或者用户看到的

4、实现数据之间的隔离。

那既然每个阶段都会涉及到不同的数据库和其他服务，要怎么实现呢？

一般我们都是通过配置文件，不同的阶段对应的配置文件的后缀不同。

前端多环境修改

Vue的process.env会保存我们的执行环境，开发就是development，生产就是production，我们可以在项目中用process.env.NODE_ENV来获取当前运行的环境，然后根据环境来判断请求后端的接口的前缀，

开发环境：localhost:8080

线上环境：user-backend.code-nav.cn

前端启动方式对process.env.NODE_ENV变量的影响：

开发环境：npm run start(本地)，process.env.NODE_ENV = development

线上环境：npm run build(会对js、vue那些代码进行打包、压缩成死的代码，也有可能是别的框架代替了npm run build命令，如umi，执行的就是 umi build)，执行完之后会出现一个dist目录就是压缩、加密后的结果。

process.env.NODE_ENV = production

要运行dist目录下的文件，就需要利用 npm 的工具serve，没安装的话要使用 npm i -g serve，安装完成之后直接到dist目录下 执行serve命令。在我们自己电脑上运行就相当于我们自己的电脑就是一个服务器，这个项目就是以production环境运行的。

不同的环境区分通过，config.xxxx.js来区分，config.development.js、config.production.js

后端多环境配置

后端多环境配置比较简单，只要在yml配置文件的后面用 - dev、-pro、-test区分即可，然后在application.yml中激活对应的版本
