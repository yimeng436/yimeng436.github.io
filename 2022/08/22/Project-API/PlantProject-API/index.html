API

介绍

这是一个API开放平台的项目，什么是API开放平台？

API开放平台实际上是给用户提供一个第三方的一个功能，如二维码解码、随机头像、每日一句话….这样的东西。

用户直接点击某个功能就会调用到我们后台的接口，给用户返回一定的数据。

这个听起来好像和我们普通的项目好像没什么区别，但是实际上需要考虑的点还是挺多的。

  • 防止攻击
  • 不能随便调用
  • 流量控制、限制次数
  • API接入时能够直接复用上面的那些保护机制

这个项目主要要实现的功能有：用户登录、开通权限、接口使用、次数统计。

管理员可以上线、下线接口、可视化接口调用情况、数据。

经过上面的介绍，我们大概知道这个项目可以分为两个端，一个是用户端、一个是管理员管理的后台管理系统。

其实基本的业务需求也就是上面的这些东西，但是我们还需要考虑一个问题就是，我要怎么实现统计调用次数，判断用户是否有权限调用、怎么对接口进行保护、并且当接口挂了这些功能 最好能够不受影响。也就是说我们需要一个集中式的保护接口的一个东西，这个东西就是API网关。

这个项目会使用微服务架构进行搭建，所以目前为止我们的项目大体上就会分成三个子模块：网关、用户、管理员。为了让用户更方便的调用接口，我们还会开发另一个模块就是构建第三方调用的SDK，让开发者少写代码就能够调用到具体的接口。

SDK最简单的来讲就是提供给用户的一个工具类，用户可以直接查看SDK的实例代码、接口文档，来使用SDK提供的接口的功能和使用方法。这个项目中也会详细介绍如何制作自己的SDK。

思考：这样的方式真的安全嘛？

显然这种做法是很不安全的：如果请求被拦截，SK就会直接被获取得到，拦截者可以用AK和SK自由访问，所以SK和密码一样一定不能够在服务器之间传输或者以明文的形式传输，这个非常重要！！！！

因此我们需要进行加密的操作，要怎么进行加密会更好呢？

如果只对SK进行加密然后再传输，这样做实际上意义是不大的，因为仅仅是加密服务器没办法知道你是真正的用户还是拦截者发来的SK密文，拦截者一样可以一直调用接口。

但是加密还是必须的，并且不能够让SK在服务器之间传递，我们需要有一个方法能够让服务器知道某次访问是谁，或者是能够标识这是一次有效的访问。

那既然要标识访问是有效的，那么服务器端就肯定需要存储这个标识，但是这个标识又不能够无限的增长，最好是定期能够被清理。

以上就是我们要实现一个比较完成的API签名认证需要考虑或者实现的功能，下面分析一下如何实现：

首先SK不能够传递，并且SK需要被加密。

一般我们会将SK和其他部分东西加在一起做一个不可逆的加密(如：MD5)，因此我们有能力的话可以自己实现一个加密算法，经过加密的结果我们成为sign，服务器端使用同样的逻辑，将数据库存储的SK重新加密，和请求过来的sign进行比对。

加密和校验实现了，我们要如何保证用一个标识来记录某次访问的有效性，最简单的办法就是通过随机数，每次请求都会带上随机数，服务器端会存储这个随机数，如果有拦截者重播请求，随机数就会失效，主要也是现在重播请求非常的容易：

但是服务器端如何保存随机数，并且让随机数在一定时间后失效呢？

1、使用Redis存储随机数，并且设计过期时间

2、每次请求都可以传递一个时间戳，随机数保存在服务器本地的map中，key为随机数，时间戳为value，每次获取对比时间戳是否在时间范围内。

API网关

上面我们已经实现了能够在不同项目之间以SDK的形式，从后端调用我们自己构建的SDK通过http请求到模拟接口项目中的接口了。

理论上我们需要完成的工作已经完成了，我们不就是希望提供一个前端，前端可以调用某些方法，后端作为一个类似于nginx的反向代理的工具，通过我们制作的SDK去调用真实提供服务的地址的接口。

但是我们也知道，这种开放式的API接口平台肯定是需要对后面的接口进行保护的，最起码我们得能够统计某个用户调用了哪个接口，最好是每个用户的调用次数不是无限的，所以我们需要制作一个用户和接口的关系表，统计用户有哪些接口的调用资格，和剩余次数，每次调用有资格调用的接口的之后，调用次数+1和剩余次数-1；

好了现在问题来了，如果我们有很多提供出去的接口，难道我们要在每次提供接口的项目中手动调用一次跟新方法吗？

这样做不仅麻烦而且还违背程序高内聚低耦合的要求，我们提供接口的模块最好只负责提供接口提供的功能，校验和统计次数这样的业务最好都不要出现在接口模块中。

当然也很容易想到使用AOP来实现，AOP是一个非常好的实现方式，但是我们可能会有别的保护接口的功能拓展，如果使用AOP可能会使得切片类很笨重，不易扩展。 还有一个比较重要的问题就是，AOP是存在于单个项目中的，而我们分为了三个项目，后端，sdk，接口模拟，处理业务的都在后端中，因此所有的entity、mapper、service都在这里面，接口模拟只负责提供接口服务，所以使用AOP没办法在接口模拟模块中拿到对应的service，当然可以使用Http、RPC来调用。

但是我们再细想一下，如果我整个模拟接口不止有一个模块呢，而是分不同的模块提供不同的接口和服务呢？

那这样每个模块我都要去编写一个AOP，这样无疑会使得项目变得冗余。

所以回想梦开始的地方，没有加一层解决不了的问题，我们直接将保护接口抽象出来单独做一个模块，这个就是API网关。

这种情况主要也是因为gateway的责任链的模式，他只有等所有过滤器执行完成后才回去执行需要被路由到的目的地址，所以也就是说，如果我们在chain.filter(exchange); 之后写的比如说要对调用成功的接口的调用次数减一，是有先后问题的，因为你这个过滤器没有结束，自然还没有调用我们真实的接口，所以直接对调用次数进行修改可能会出现逻辑上的错误，因为此时还不能知道真实的接口调用是否会出现问题。

但是如果我们不在后面增加修改调用次数的逻辑，又好像没地方加了，又不能写在真实接口上，又不能写在过滤器里面，怎么办。

RPC

上面我们在网关模块还留了一些 待完成的工作，就是我们还要判断调用的接口是否存在以及需要操作数据库，让用户的总的调用次数和剩余调用次数发生变化。这些都是要跨项目调用的问题。

一般跨项目或者说跨模块之间的调用无非就是 http和rpc了，这里我们使用RPC来调用，上面使用过了Http来完成远程调用的工作了。

Http和RPC区别，这也是面试常问的一个问题

Http请求，我们需要使用的时候一般都需要使用一些第三方的封装了Http请求的包，比如说Hutool，并且我们还需要为了调用这个接口，封装好各种参数，写好请求头等信息，其实没有那么方便，就是并不能够和调用我们本地的代码一样只需要直接调用即可。

相比于Http，RPC会相对更加透明一些，它的目标就是让使用者调用不同的项目中的方法的时候能够和调用本地方法一样简单方便。

当然说到这里，肯定有人会说Http同样能够被精简得像调用本地方法一样调用了，比如常用的feign和openfeign，这些都是对http请求的封装，能够非常简介的完成模块之间的调用。

但是RPC最独特的地方就在于RPC可以不使用Http请求作为发送的协议，如果项目觉得http协议的七层协议效率太差的话，完全可以只使用原生的TCP/IP协议实现RPC，从而达到更好的性能

我们这里使用dubbo，dubbo底层就不是Http协议，而是性能较高的Triple协议。

Dubbo使用也非常简单，其实和Feign那些挺像的，都是通过服务暴露出接口之后，消费方直接调用。

服务方通过向注册中心注册自己，消费方向注册中心请求的时候，注册中心只是告诉消费放这个服务是由谁提供的，而不是通过注册中心调用服务提供方。

1、简易的封装了Http请求，通过spring.factories配置文件，配置了我用于封装Http请求的客户端，制作了一个简单的starter供开发者方便调用。并且通过spring-boot-configuration-processor这个依赖，让idea工具在yml配置文件中，能够识别我的starter的配置参数。

2、简单了解了API签名认证，并且以比较规范的方式实现了一遍，通过AK和SK来完成API签名认证。由于SK一定不能够在服务器之前传输，所以在前端传递给后端模块后，经过加密得到一个sign，通过加密后的sign来作为http请求的身份识别标志。 http请求模拟接口模块的时候会带上AK和sign，模拟接口模块会利用AK从数据库中查出SK，利用相同的加密逻辑，得到一个真正的sign，和请求携带的sign进行比对，只有相同才有资格调用接口。

​ 但是，为了防止请求被恶意回放，加上了一个随机值，校验放会存储这个随机值，如果已经存在也不允许调用接口，但是为了防止随机值沾满内存所以想了两种方式实现：

  • redis，缓存这个随机值，并且设置过期时间
  • 利用布隆过滤器，尽量减少内存的开销，但是会出现一定的误判率，在超大流量的情况下感觉可以用这种方式

3、为了统一管理接口的后续操作，比如说日志打印、统计调用次数，或者对接口进行统一的保护，也是想了三种方式实现：

  • AOP，这是一开始想到的方法，因为如果统计次数这样的需求，如果提供一个接口就调用统计次数的方法，太过冗余。所以用AOP这种非侵入式的方法来实现，次数统计以及权限校验。 但是后面想了想如果开发模拟接口的不止同一个项目的话，每个项目都要拷贝一份AOP，或者单独实现一个AOP，还是会出现一定冗余。

  • API网关，最后选用的是网关的方式来完成对接口的统一管理和保护。因为单独抽象出一层的方式，能够极大的减少冗余，并且很方便易扩展，如果有多个模拟接口的项目那也只需要在网关处配置多一个路由就能够完成。

    实现过程遇到了一个比较大的麻烦，就是全局过滤器中，一开始以为chain.filter(exchange)这个方法，是阻塞式的，能够在调用完真实接口之后才会返回，这样的串行方式就能够很容易的在chain.filter(exchange) 代码后面写上我统计接口调用次数的逻辑。

    但是很不幸，chain.filter(exchange)方法是异步的，并且整个gateway会等到所有过滤器都正常返回后才会调用真实接口，所以直接串行的进行统计接口调用次数，逻辑先后上会出现非常明显的问题。

    解决方法：通过使用response的装饰器对象，并且重写writeWith方法，并且使用这个装饰器对象传递给后续的过滤器，让接口执行完成后，会将返回结果写到请求体中，自然会调用writeWith方法，从而实现了在接口执行完成后，调用我们的处理逻辑。

  • MQ异步处理，这种方式主要是用于接口调用完成之后的后续操作处理的一个补充，因为考虑到后续业务逻辑如果较多、或者日志记录的内容较多，可能会导致整个系统性能下降，从而采用异步对后续进行处理，让接口返回值能够尽早的响应给用户，而其他用户不需要实时获取的数据，则在后台慢慢执行。

4、在统计调用次数、鉴权部分使用了RPC技术，使用dubbo来完成对业务类接口的复用。
