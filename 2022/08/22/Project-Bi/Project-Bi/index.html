智能数据分析

需求分析

我们希望将数据分析这件事情交给AI来完成，这样我们只需要提供用户的数据，用户将数据给AI之后，AI会负责将数据绘制成图标，并且提供一段对这段数据的分析，从而减少人力的开销。

核心内容

对接AI

我们现在市面上的那些ai问答系统基本上再一次对话都是有词数限制的，所以我们如果能用更少的词数得到相同的答案这样更能够节省成本。因此我们需要对用户的输入进行处理：

  • 数据压缩：用更少的词进行提问，交给AI服务
  • 关键词提取：提取提问的关键词，同样是减少词数

我们现在用户会上传一个excell文件，我们可以先对excell文件读取后进行预处理，转换为占用更小的存储格式CSV。下面是CSV和excell的对比

可以看到csv 几乎就可以当作是String的存储。

其实就是简单的使用一个easyexcel的第三方的工具库。

我们把传递的excel数据，转为这样的string，然后传给AI，并且我们传的时候还可以加上我们要求AI分析的需求(目标)。

我们对接完AI接口后，需要考虑几点问题。

  • 问题1：现在我们数据库的设计是用一个字段来存储，用户上传的原始数据，当用户上传数据特别大的时候，会导致整张表的查询变得很慢，如何解决？

  • 问题2：用户绕过前端调用后端接口疯狂刷次数怎么办？

  • 问题3：AI生成速度太慢怎么办？

问题1

现在我们数据库的设计是用一个字段来存储，用户上传的原始数据，当用户上传数据特别大的时候，会导致整张表的查询变得很慢，如何解决？

最简单的解决方法就是，用一台单独的文件服务器去存储用户的原始数据，然后数据库改为存放服务器数据的链接。

这种方式能够降低数据库的压力，并且用户数据也不会互相影响，拉低整个数据库的查询速度。

但是会带来网络传输的问题，如果用户数据过大，可能会导致数据网络传输速度很慢，以及服务器性能需要比较高，因为可能会有频繁的磁盘IO。

第二种方式，动态的为每个数据创建一个表，用户每个原始数据对应一个表，这样用户查询原始数据的时候，就可以和查询数据库一样使用简答的数据库查询语句来得到用户的数据。

用户长传之后我们根据上传得到的表记录的ID，拼接其他的值如：char_{表记录ID}，作为数据库的表名。

然后遍历用户数据的表头，拼接作为表的字段名。然后遍历插入数据。

这样做的好处，可以隔离用户数据之间的影响，不会让一个用户的数据过大导致拉低整个表的查询效率，用户之间的数据的隔离的。但是这样做可能会导致数据库的表过多。不容易维护

两种方式各有利弊。

问题2

用户绕过前端调用后端接口疯狂刷次数怎么办？

说白了就是需要做一个限流，防止恶意攻击。

在一些大并发的情况下，比如商城网站活动促销的时候，有可能你的请求连后端每过去，前端做了一个随机限流，就已经把你的请求拦下来的。

但是这并不意味着我们可以不要再后端进行限流，因为肯定会有人通过某些手段绕过前端发送请求的。所以后端限流也是必要的。

限流算法种类

主要分为四大类：

  • 固定窗口

固定一个时间段内只运行处理固定个数的请求。如固定一小时内只允许处理10个请求，这里的固定窗口就体现在在，我是固定第一个0到60分钟只能处理10个，第二个0-60处理另外10个。

这样做实现起来比较简单，但是容易出现流量突刺的问题，比如我在第一个时间段的第59分钟来了10个请求，这10个请求还没有处理完，但是已经刷新可以处理请求的时间段了，如果我在第二个0-60时间段内又来了10个请求，也就是2分钟要处理20个请求，这样就可能会导致短时间内大量的请求到达服务器。

  • 滑动窗口

和上面的类似，只是从自然时间段变成了，从第一个请求到来的时间开始计算到什么时候才刷新能够处理的请求次数。比如说：59分中到达了请求，那请求次数刷新的时候就变成了1小时59分，才会刷新能够处理另外的10个请求。

一定程度上解决了流量突刺，但是这个滑动窗口的值不好设定。

  • 漏桶限流

以固定的速率处理请求，比如说允许10分钟处理10个请求，就是一分钟会处理一个请求。如果我顺序到达了10个请求，但是第一个请求还在执行，此时第11个请求到达，则请求被拒绝。

这样看似非常的自然能够实现限流，但是每个请求之间是串行的，10个请求就算请求成功了，第十个请求也要等10分钟才能够被处理。用户体验很差

  • 令牌限流

其实这种方式是可以是上面几种方式的结合，因为他需要服务器规定一段时间会生成指定个数的令牌，请求会去获得令牌，获得令牌的请求，才能够被处理。如果没有获得令牌的请求直接被拒绝。

所以需要确定多久生成令牌，相当于是第一、第二种方式的问题；拿到令牌的请求是可以被并发处理的，这样的话相比于漏桶限流算法的并发度和用户体验会好一点。

限流的实现

一般我们会利用第三方的限流库来帮我们实现限流。

  • 单机版

只有一台服务器，那当然就只要单机实现就好了，一般会使用Guava为我们提供的RateLimiter类来实现

  • 分布式版

Redisson，为我们提供了分布式下的流量限制的工具，同样是RateLimiter。

问题3

AI服务生成速度太慢怎么办？

很明显这是一个顺序执行的问题。

这类问题基本上就可以归结为：我们的代码中有部分业务逻辑非常耗时，用户获得响应的时间需要等待很久，并且用户可能获得非预期的结果，所以为了提高用户体验绝对不能让用户在当前状态下等待。

解决这类问题，基本上都是使用异步化来解决。

标准异步化业务的流程

1、把用户的提交当作是一个任务，不让用户在当前页面一直等待，而是提交了一个任务后，用户可以看到自己提交的任务的当前状态，用户任务执行完成之后会有一系列的提示（系统弹窗、短信、邮箱等）。

我上面一直用的是任务，所以应该很容易可以联想到，我要说的是 Thread。

当用户提交任务时候我们需要考虑：

  • 提交任务成功
    • 当前有空闲线程，直接执行用户任务
    • 如果没有空闲线程，将任务放到任务队列里面
  • 提交任务失败，任务队列都满了/网络异常等情况
    • 首先我们必须要对这些任务进行持久化无论成功还是失败，尤其是失败更应该持久化。一来是我们可能有必要对这些失败的任务进行分析失败的原因；二来是可以让用户对于这些失败的任务再进行重新提交、或者当系统空闲的时候系统自己对这些任务选择性重试。

当然异步是一个非常复杂的东西，因为很多东西是不可控的，我们甚至没办法知道什么时候程序运行到什么个状态了。所以能用同步的还是一定用同步，不得已才使用异步，并且对于一个十分复杂的异步任务，最好划分为多个小任务，每个小任务执行结束能够做出一些提示、甚至断点，失败后从上一个任务开始。

上面说了那么多需要考虑的问题，这些东西如果要自己实现，那要考虑的东西又更多了，比如说：

  • 我应该定义多少个线程
  • 多出来的线程怎么处理
  • 线程不够需不需要增加
  • 队列满了、具体怎么执行
  • 线程执行完了，怎么继续执行下一个
  • 怎么让线程知道还有任务没有执行
  • …..

这些都是需要我们自己去实现对线程的管理，但是这些东西 大部分编程语言也都帮我们实现好了，那么就是线程池。

Java中一般建议使用 JUC自带的线程池的实现 ThreadPoolExecutor 。

相关配置见：线程池参数配置

使用线程池之后，上面需要考虑的东西只需要创建线程池的时候配置一下就行了。

使用线程池之后，我们就可以把我们的原本同步的代码，用CompletableFuture 构建为异步的了。

主要就是把 调用AI服务那一块改为异步实现。

并且使用我们自己的线程池，这样任务的提交，线程的管理，任务队列的管理都会由线程池完成。

异步化之后的问题

我们上面 通过线程池，实现了单机的异步化，但是如果我们服务器变多，每个服务器都配置了核心的线程数，那就有可能会导致，一次性AI服务要处理的请求高于承载能力。

这个问题其实是分布式正常都会存在的问题，比如说本地锁没办法在分布式场景下起作用也是一样的问题。

所以解决这个问题，一般都是通过加一层，集中式的存放不同服务器之间需要共享的数据。

这个中间层一般也都是 redis。

除了上面这种问题以外，我们上面的任务队列是基于内存的，所以如果服务器出现宕机或者异常，整个任务队列的内容都会丢失，虽然可以通过在数据库里面恢复，但是这样就要单独去为异常情况编写不同的代码。

而且这种重试机制，应该是很多系统通用的，所以可以用一个通用的解决方式，那就是消息队列。
